MOPP code data for collision h0001 in the Depths (bonfire room), which has 835 vertices.

This data represents a hierarchical bounding volume tree (BVH). The "leaves" of this tree
will almost certainly be individual triangle faces in the collision mesh, probably by
indexing into the `indices16` property of the mesh, which contains face definitions in
TRIANGLE_STRIP form. Non-leaf nodes of the tree will be larger volumes that contain both
the nodes below them.

Based on what I know about MOPP, it strikes me as likely that it does literally nothing
else except define these bounding volumes and the overall tree structure. We also know
it's a BINARY tree, so each node has at most two children (and probably never one child,
in practice). This gives the collision detection logarithmic complexity in the number of
mesh faces (leaves). Although, the leaves do not have to be single triangles, and likely
are not - but they could be a primitive collection of triangles...

The BuildType for this MOPP code is "BUILT_WITH_CHUNK_SUBDIVISION" (0), and I believe the
chunk size is always 512, so that's a clue. But there may also be a header.

	/// Defines the size of each mopp chunk (512 bytes)
	#define HK_MOPP_CHUNK_SIZE 512
	#define HK_MOPP_CHUNK_MASK (HK_MOPP_CHUNK_SIZE - 1)
	#define HK_MOPP_LOG_CHUNK_SIZE 9

There's also talk of a "range of property values", which seems to be just any unsigned
integer, and "number of property values", which seems to be 1 (max). Finally, each
`hkpMoppCode` object has a Vector4 `offset` member, which likely just places the BVH in
the world (e.g. at the center of the room or whatever).


From nitaigao:

	The MOPP-tree is an optimised polytope tree, based on the same principles as KDOP
	trees. In order to keep the memory down, the MOPP data structures:
		- use a reduced floating point resolution to store the extents of each MOPP node.
		- store the minimum number of extents for each node.
		- implement various compression techniques to reduce the overall size.
		
I don't like the sound of "various compression techniques". Hopefully they're just
overselling it. (An "extent", by the way, is just a serialized data section, I think?
Or it might be the basis for the node's shape, based on the previous line.)

Good news is that I don't think the compiler inputs will help me decode it; they affect
the actual non-leaf shapes that are created (and if I ever want to WRITE these codes, I'll 
need something there) but that's all.
